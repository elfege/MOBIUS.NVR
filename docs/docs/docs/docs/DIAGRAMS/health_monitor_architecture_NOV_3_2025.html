🚨 Critical Issues Identified

Issue #1: Health Monitor Never Reattaches After Failed Restart

⚠️ CRITICAL BUG: Once a restart fails, the health monitor is permanently detached and never reattaches, leaving the stream in a broken state with no automatic recovery possible.
Problem Flow:
  1. Health detects stale frame → schedules restart
  2. restartStream() called → DETACHES health monitor (line 495)
  3. forceRefreshStream() throws network error
  4. Catch block sets status to "Restart failed"
  5. Health monitor NEVER REATTACHES ❌
  6. Camera stuck forever - no more auto-retries possible
📍 Location in Code: stream.js lines 488-534 
async restartStream(cameraId, $streamItem) {
    try {
        // ... restart logic ...
        
        // Detach health monitor during restart (line 495)
        if (videoElement && videoElement._healthDetach) {
            videoElement._healthDetach();
            delete videoElement._healthDetach;
        }
        
        // If forceRefreshStream() fails here...
        await this.hlsManager.forceRefreshStream(cameraId, videoElement);
        
    } catch (e) {
        // ...error is caught here
        console.error(`[Restart] ${cameraId}: Failed`, e);
        this.setStreamStatus($streamItem, 'error', 'Restart failed');
        // ❌ NO REATTACHMENT OF HEALTH MONITOR!
    }
}

Issue #2: Attempt Counter Persists But Monitoring Stops

Inconsistent State: When restart fails:
  • ✅ restartAttempts.set(serial, N) persists
  • ❌ restartTimers.delete(serial) was already called
  • ❌ Health monitor is detached
  • Result: System thinks it attempted N times, but won't try again

Issue #3: No Backend Stop = Potential Zombie Streams

Frontend-Only Stop After Refactoring:
  • Previous behavior: Called /api/stream/stop/ endpoint
  • Current behavior: Client-side only (HLS.js destroy, img src clear)
  • Good: Multiple clients don't kill each other's streams
  • Issue: When frontend refresh fails, backend stream keeps running
  • Issue: Health monitor can't distinguish "stream frozen" from "network error"

💡 Proposed Solutions

Solution #1: Reattach Health Monitor on Restart Failure (Recommended)

Approach: Add health monitor reattachment in the restartStream() catch block. 
catch (e) {
    console.error(`[Restart] ${serial}: Failed`, e);
    this.setStreamStatus($streamItem, 'error', 'Restart failed');
    
    // ✅ REATTACH health monitor even on failure
    const el = $streamItem.find('.stream-video')[0];
    if (el && this.health) {
        if (streamType === 'HLS' || streamType === 'LL_HLS' ...) {
            const hls = this.hlsManager?.hlsInstances?.get?.(serial);
            el._healthDetach = this.health.attachHls(serial, el, hls);
        } else if (streamType === 'MJPEG' || streamType === 'mjpeg_proxy') {
            el._healthDetach = this.health.attachMjpeg(serial, el);
        } else if (streamType === 'RTMP') {
            const flv = this.flvManager?.flvInstances?.get?.(serial);
            el._healthDetach = this.health.attachRTMP(serial, el, flv);
        }
    }
}

Solution #2: Don't Detach Health During Restart

Approach: Keep health monitoring running during restart attempts. Only detach/reattach when explicitly stopping.
Pros: Simpler logic, continuous monitoring
Cons: May trigger false STALE warnings during restart

Solution #3: Add Retry Logic in forceRefreshStream

Approach: Wrap HLS startup in retry logic with exponential backoff.
Pros: More resilient to transient network errors
Cons: Adds complexity to stream manager
