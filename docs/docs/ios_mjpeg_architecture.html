iOS MJPEG Streaming Architecture

Last updated: January 6, 2026 02:45 EST

The Core Problem

iOS Safari has a hard limit of ~4-8 simultaneous video decodes.

With 11+ cameras using HLS (which requires <video> tags), iOS simply can't decode them all. Some streams show black screens or fail to load.

Solution Attempt: Use MJPEG for iOS grid view. MJPEG uses <img> tags which have no decode limit - Safari can display hundreds of images.

What WORKS on iOS

  • Reolink cameras with native MJPEG (Terrace South, REOLINK OFFICE) - Direct HTTP connection to camera
  • HLS streaming - Works but limited to ~6 streams max
  • Sub-second latency on desktop - LL-HLS via MediaMTX achieving ~300ms

What's BROKEN on iOS

  • MediaServer MJPEG - Eufy, SV3C, and other single-connection cameras
  • Clients connect but disconnect before receiving frames
  • Frame buffer race condition (fixed in latest commit, needs restart)

3. The Dependency Chain Problem

┌─────────────────────────────────────────────────────────────────────────────┐
│                    MediaServer MJPEG Dependency Chain                       │
└─────────────────────────────────────────────────────────────────────────────┘

    For MediaServer MJPEG to work, ALL of these must be true:

    1. Camera must be online and responding to RTSP
       └── Eufy cameras only allow ONE RTSP connection!

    2. Dual-output FFmpeg must be running
       └── Started by /api/stream/start/{camera}
       └── Takes 2-3 seconds to initialize

    3. FFmpeg must be publishing to MediaMTX
       └── Sub stream at rtsp://nvr-packager:8554/{camera}
       └── Main stream at rtsp://nvr-packager:8554/{camera}_main

    4. MediaServer MJPEG service must connect to MediaMTX RTSP
       └── Runs FFmpeg to decode H.264 → MJPEG
       └── Takes 1-2 seconds to start

    5. Frame buffer must have a frame ready
       └── "Connecting..." frame set immediately ← BUG WAS HERE
       └── Real frames arrive after FFmpeg produces output

    6. HTTP generator must yield frames to iOS client
       └── iOS Safari may timeout if no data arrives quickly

    ════════════════════════════════════════════════════════════════════
    TOTAL STARTUP TIME: 4-6 seconds before real video appears
    iOS Safari timeout: Unknown, but seems to disconnect quickly
    ════════════════════════════════════════════════════════════════════

4. Timeline of iOS MJPEG Attempts

Jan 5, 22:15 - Identified iOS video decode limit

iOS Safari can only decode ~4-8 simultaneous <video> elements. HLS uses video tags, so not all streams load.

Jan 5, 22:30 - iOS HLS fallback from WebRTC

WebRTC doesn't work on iOS (requires DTLS-SRTP encryption). Added fallback to HLS for iOS WebRTC cameras.

Jan 5, 23:00 - Sequential stream loading

Added 300ms delay between stream starts. Helps but doesn't solve the 4-8 stream limit.

Jan 5, 23:30 - MJPEG grid for portable devices

Force MJPEG (<img> tags) for iOS/Android grid view. No video decode limit! But need MJPEG source for all cameras.

Jan 6, 00:30 - MediaServer MJPEG service created

For cameras without native MJPEG (Eufy, SV3C), tap MediaMTX RTSP and convert to MJPEG via FFmpeg.

Jan 6, 01:58 - MJPEG third FFmpeg output (FAILED)

Tried adding MJPEG as third output to dual-output FFmpeg, publishing to MediaMTX. MediaMTX rejects MJPEG over RTSP. RTSP doesn't support MJPEG codec well.

Jan 6, 02:12 - Reverted to MediaServer tap approach

MediaServer MJPEG taps H.264 from MediaMTX, decodes with FFmpeg, serves MJPEG. Capture loop works but clients disconnect.

Jan 6, 02:14 - iOS pagination (6 per page)

Limit to 6 cameras per page on iOS. Stops hidden streams. Reduces load on Safari.

Jan 6, 02:30 - Frame buffer race condition found

Logs show "No frame available on first request" - frame buffer empty when HTTP generator starts. Race between add_client() and capture thread.

Jan 6, 02:40 - Fix: Pre-initialize frame buffer

Commit 656d98a - Set "Connecting..." frame in add_client() BEFORE starting capture thread. Needs container restart to test.

7. What To Try Next

Immediate (After Container Restart)

  1. Test if "Connecting..." frame appears on iOS (commit 656d98a)
  2. Check logs for "Sending frame #1" messages
  3. If still broken, add more timing delays

Alternative Approaches If MediaServer MJPEG Continues Failing

  1. Snapshot polling - Don't use multipart, just set img.src to snapshot URL with timestamp every 500ms
  2. Native HLS with fewer streams - Accept 6-stream limit, use pagination
  3. Server-Sent Events (SSE) - Different protocol than multipart, might work better with iOS
  4. WebSocket image delivery - Binary frames over WebSocket
